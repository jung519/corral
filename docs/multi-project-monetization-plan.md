# 작업계획서 — 멀티 프로젝트 + 온라인 엔타이틀먼트 유료화

> 계획서(코드 없음). 사업 판단은 사용자 결정을 반영하되, 남은 미결(오픈소스 여부)은
> 추천과 함께 열어 둠.

## 1. 목적
회사 생활처럼 **한 사람이 여러 프로젝트**를 담당할 수 있게, 상위(홈/조직) 화면에서
프로젝트를 추가·선택 → 선택 시 현재 Corral 화면. 그리고 **여러 프로젝트 사용을 유료**로.

## 2. 확정된 수익화 모델 (사용자 결정)
- **로컬 실행 유지**: 클라우드 호스팅은 논외. 에이전트는 로컬에서 그대로(BYOK).
- **무료**: 1개 프로젝트. 오프라인·무로그인 가능.
- **유료**: 2개+ 프로젝트 → **인터넷 연결 + 로그인 + 유료 엔타이틀먼트 필수.**
- **타깃**: 개인(B2C) + 회사/팀(B2B) 둘 다.
- **백엔드**: 인증·과금·서버 운영을 솔로로 감당할 의향 있음(단, 가벼운 백엔드 지향).

성격 요약: **클라우드 서비스가 아니라 "로컬 앱 + 온라인 라이선스/엔타이틀먼트 게이트".**

## 3. 기술 A — 멀티 프로젝트 구조
현재는 단일 프로젝트 전제(전역 상태 1개). projectId로 네임스페이스:

| 부품 | 지금 | 멀티화 |
|---|---|---|
| config | 단일 파일 | 프로젝트별 |
| 시크릿(키체인) | `anthropic:default` | `<projectId>:anthropic:default` (충돌 방지) |
| 상태/이력 | 단일 디렉터리 | 프로젝트별 |
| 워크스페이스 | 단일 루트 | 프로젝트별 |
| — | — | **프로젝트 레지스트리 + 전환 UI(홈 → 선택 → Corral 화면)** |

- 독립 설정 = 네임스페이싱의 결과. **가져오기** = 다른 프로젝트 config 복사(시크릿은 참조/재입력).
- **동시성**: MVP는 한 번에 한 프로젝트 활성(전환 시 오케스트레이터 재기동) 권장.
- 규모: 솔로 가능·중간. 난점 = 키체인 시크릿 네임스페이싱 + 전환.

## 4. 기술 B — 로그인 / 엔타이틀먼트 게이트
- **인증**: 로그인(이메일/OAuth) → 세션 토큰.
- **엔타이틀먼트**: 서버가 "이 사용자: free(1) / pro(N)" 반환.
- **게이팅 지점**: **2번째 프로젝트 생성/전환 시** 온라인 로그인 + pro 권한 확인. 1개는
  무로그인·오프라인 허용.
- **오프라인/유예**: 권한을 캐시하고 주기적 재검증. 오프라인이면 유예창 후 멀티 잠금(UX 설계 필요).
- **백엔드(가벼움)**: 인증(Clerk/Auth0/Supabase/自前) + **Stripe**(구독) + 웹훅 → 엔타이틀먼트
  갱신 + 권한 조회 엔드포인트. 호스팅 에이전트 실행 없음 → 인프라·비용 작음.

## 5. 핵심 미결 — 오픈소스 여부와 "강제력" (⚠ 유일한 관건)
> **로컬 유료 게이트의 강제력은 앱이 오픈이냐 클로즈냐에 전적으로 달림.**
- **오픈소스면**: 게이트 코드 제거·재빌드로 우회 쉬움 → 유료화 명목상. (선의·B2B 컴플라이언스엔 통함)
- **클로즈드면**: 바이너리 패치 필요 → 마찰↑, 대다수 억제. B2C 정직·B2B 준수엔 유효.
- **추천: 오픈코어 분리** — **코어 엔진은 오픈**(신뢰·채택), **데스크톱 셸(로그인·엔타이틀먼트·
  멀티프로젝트)은 클로즈.** 오픈 신뢰 + 강제력을 동시에. (기존 "오픈코어" 방향과도 일관)

→ 이 결정이 서면, 나머지 설계는 확정 가능. **미결: 오픈 유지 / 클로즈 전환 / 오픈코어 분리(추천).**

## 6. 리스크
- **마찰**: 개발 툴에 강제 로그인·인터넷은 특히 B2C/OSS층 저항. 완화 = 무료 1개는 완전 오프라인.
- **오프라인 UX**: 인터넷 끊길 때 멀티 프로젝트 잠금 처리(유예·안내) 설계 중요.
- **오픈이면 우회**: 위 §5. 오픈코어로 해결 권장.
- **운영 부담**: 인증·과금·개인정보·지원의 지속 비용(가벼워도 0은 아님).

## 7. 단계
1. **멀티 프로젝트(로컬)**: 네임스페이싱 + 홈/전환 UI + 가져오기. (유료화 전, 로컬만)
2. **인증·엔타이틀먼트 백엔드**: 로그인 + Stripe + 권한 엔드포인트.
3. **게이트 연결**: 2번째+ 프로젝트에서 로그인·권한 확인, 오프라인 유예.
4. **오픈/클로즈 구조 확정**(§5) → 배포 파이프라인.
5. 결제·요금제·법무(약관/개인정보) 마감.

## 8. 솔로 적합성
- §3(멀티) 솔로 가능·중간. §4(게이트) 가벼운 백엔드라 솔로 가능하나 **인증·과금·법무가 새 영역** →
  학습·운영 비용 발생. 전체적으로 **가능하되 §3+§4는 별개 큰 덩어리**라 순차 진행 권장.

## 9. 비범위
- 클라우드 호스팅 실행·동기화(사용자가 논외로 함).
- 실제 코드 작업(이번 세션은 계획만).

## 10. 미결 결정
- **(핵심) 오픈소스 구조**: 오픈 유지 / 클로즈 / **오픈코어 분리(추천)**.
- 요금제(무료 1 / pro N개·가격), 오프라인 유예 기간, 인증·과금 스택 선택.
