# 작업계획서 — 조직 의도/방향 주입 (Direction Injection)

> 이 문서는 **계획서**입니다. 코드 작업은 포함하지 않습니다.

## 1. 목적
Corral이 "외부 도구를 잇는 오케스트레이터"를 넘어서, **AI가 회사의 목적·방향을 아는
직원처럼 이슈를 판단·계획·리뷰**하게 만든다. 같은 문제라도 "회사가 원하는 방향"으로
풀어내는 것 — 이것이 단순 연결형 도구와의 **차별점(해자)**.

## 2. 배경 / 현재 구조
현재 AI 추론에 들어가는 작업자 측 입력:
- `profile { language, stack }` — 얕은 맥락.
- **skills(reference repo)** — 컨벤션·디자인시스템. **이미 계획·구현·리뷰에 주입**됨
  (WORKFLOW "Skills/conventions REQUIRED", plan critique·review 프롬프트에서 compliance 확인).

빠진 것: **"의도/방향"** — 회사 목적, 제품 우선순위, 트레이드오프 판단 기준, 작업자 성향.
→ 계획의 방향·옵션, 리뷰의 보정에 **작업자 의도가 실리지 않음.**

## 3. 핵심 개념 — 3개 층 분리
| 층 | 답하는 질문 | 성격 | 상태 |
|---|---|---|---|
| skills | 어떻게 생겨야 하나 | 규칙(binding) | 있음 |
| profile | 어떤 스택/언어 | 맥락 | 있음(얕음) |
| **Direction(의도)** | **왜 / 어느 방향으로** | **판단 기준(guiding)** | **신설** |

Direction에 담는 것: ① 회사/제품 목적, ② 방향·우선순위(예: 안정 우선 vs 속도 우선),
③ 의사결정 원칙(트레이드오프 기준), ④ 작업자 성향(최소 변경, 의존성 회피 등).

## 4. 접근 확정: RAG ❌ / 주입 ✅
- 의도는 **작고 항상 관련** → 검색 불필요, **항상 주입**.
- skills와 동일한 주입 메커니즘, 단 프레이밍이 다름: **"comply(규칙 준수)" → "judge in line with(이 방향대로 판단)"**.
- RAG는 별개 층(큰 지식 코퍼스의 선택적 회수)으로, 이 기능 이후 필요 시.

## 5. 범위 — 저장 위치 (확정: 전역 + 프로젝트)
| 구분 | 위치 | 내용 |
|---|---|---|
| **전역(조직/작업자)** | **userData 파일 `direction.md`** (머신 단위, 모든 config/프로젝트 공통) | 공통 의도·성향 |
| **프로젝트** | `.corral/DIRECTION.md` (레포 안, 버전관리) | 이 레포의 목적·방향 |

> 전역 저장은 **(b) userData 파일**로 확정(결정 F). config 필드가 아니라 머신 단위 파일이라,
> 여러 프로젝트/여러 config를 관통하는 진짜 전역이 됨. 코어는 경로를 주입식으로 받아 읽는다
> (상태 디렉터리 `CORRAL_STATE_DIR` 패턴과 동일).

**이슈별 방향은 전용 기능을 만들지 않는다(결정 D).** 특정 이슈만 다른 방향이 필요하면
사용자가 **이슈 본문**(최초 작성 시) 또는 **검토 단계 지시**(질문·논의 팝업의 수정요청/
지시승인)에 적는다 — 두 채널 모두 이미 추론에 주입된다. 트래커 전용 필드/라벨은
스키마 종속·중복이라 배제.

판단 우선순위(높→낮): **이슈 본문/검토 지시 > 프로젝트 Direction > 전역 Direction.**
단 **이슈의 정확성·요구사항이 항상 최우선**(Direction는 동률일 때의 기본 방향).

## 6. 주입 지점 (확정: 계획 + 리뷰)
- **계획**: kickoff · plan critique · consolidate 프롬프트에 Direction 블록 주입 →
  **방향과 생성 옵션이 의도를 반영**(옵션 축을 `보수/균형/적극`처럼 성향 축으로 제시).
  **+ 이슈 본문/검토 지시의 방향 힌트를 "그 이슈에 한해 Direction보다 우선 적용"하라고 명시**
  → 별도 UI 없이 이슈-레벨 오버라이드가 동작(결정 D 보강).
- **리뷰 (결정 C)**: 프로젝트 우선순위·방향에 맞춰 심각도·판단 보정(MVP면 NIT 완화,
  성숙 프로젝트면 엄격). **개발 방향은 리뷰까지 끝난 PR(의미 단락)에서 완성되므로 리뷰에도
  동일 Direction를 적용**한다.
- **구현(선택)**: 접근/스타일.

프레이밍 가드(프롬프트에 명시):
> "Direction는 **동률일 때의 기본 방향**이다. 이슈의 정확성·요구사항이 우선한다.
> skills(규칙)는 여전히 binding이다. Direction는 방향 판단의 기준이다."

## 7. 설계 개요 (구현 시)
- 저장: 전역 = 자유텍스트 설정값, 프로젝트 = `.corral/DIRECTION.md`.
- UI: 셋업/환경설정에 Direction 입력란(자유텍스트, 결정 B).
- 주입: 프롬프트 빌더/워크플로 가이드에 Direction 블록 추가 — **기존 profile 주입과 동일 패턴**.
- 형식: **자유텍스트**(결정 B). 구조화 강제는 안 한다.

### 7.1 작성 가이드 (결정 B의 핵심)
내용은 결국 **사용자가 자유텍스트로 작성**한다. 우리는 "어떻게 쓰면 좋은지"만 안내한다 →
빈 화면을 주지 않도록 **도움말 + 플레이스홀더 + 스타터 템플릿**을 제공:
- 권장 섹션: **목적 / 우선순위 / 의사결정 원칙 / 하지 말 것**.
- 각 섹션 1~2줄 예시(예: "우선순위: 안정 > 기능 속도", "원칙: 새 의존성은 정당화 필요").
- 자유도는 유지하되, 좋은 Direction의 형태를 예시로 보여 품질을 끌어올린다.
- (Direction 품질이 결과 품질을 좌우하므로, 이 가이드 자체가 중요한 산출물.)

## 8. 단계 (전역 = userData 확정 반영)
0. **전역 저장 배선(userData)**: 코어가 userData `direction.md`를 주입식 경로로 읽음
   (`CORRAL_STATE_DIR` 패턴). 데스크톱에 읽기/쓰기 IPC 브리지.
1. **Direction 주입(계획) + 병합**: 전역(userData) + 프로젝트(`.corral/DIRECTION.md`)를
   우선순위대로 병합 → WORKFLOW.md liquid 블록 + kickoff/consolidatePlan 프롬프트에 주입.
2. **전역 Direction UI**: 환경설정에 자유텍스트 입력 + 스타터 템플릿 + §14 설명 문구 →
   userData `direction.md`에 저장(IPC).
3. **이슈-레벨 오버라이드**(결정 D): kickoff/review 프롬프트 한 줄로 이슈 본문/검토 지시 우선.
4. **리뷰 보정**(결정 C): reviewRoundPrompt+consolidateReview에 Direction "심각도 보정" 프레이밍.
5. **옵션 방향 축**(선택): plan_options 생성에 보수/균형/적극 축.
6. **프로젝트 Direction 문서/UI + eval**: `.corral/DIRECTION.md` 안내, Direction on/off 비교.

**최소 동작 슬라이스 = 0+1+2+3+4** (전역 방향이 계획·리뷰에 실리고, UI 편집, 이슈 오버라이드).

## 9. 1인 개발 적합성
- 거의 전부 **프롬프트·설정·UI** 작업 → **솔로 가능, 며칠 규모**. RAG 같은 인프라 없음.
- 가장 어려운 부분: **효과 입증(eval)** 과 **Direction 문구 튜닝**(과하지도 모자라지도 않게).

## 10. 리스크
- **과적재/모순**: 의도가 너무 많거나 충돌하면 편향·프롬프트 비대 → 간결 유지.
- **하드룰 오해**: 에이전트가 Direction를 규칙처럼 강제 적용 → "기본 방향" 명시로 완화.
- **정확성 침해**: 방향 때문에 옳은 해법을 비틀 위험 → "정확성·요구 우선" 가드.

## 11. 차별점 (왜 중요)
연결형 도구는 *"누가 일하나"* 만 정한다. Corral은 **"회사 의도대로 판단"** 을 인코딩 →
직원처럼 일관된 방향으로 계획·리뷰. 도구 교체로는 복제 안 되는 **제품 해자**.

## 12. 비범위 (이번 단계 밖)
- RAG / 임베딩 / 대형 지식 인덱싱 (별개 층, 추후).
- 실제 코드 작업 (이번 세션은 계획만).

## 13. 결정 (확정)
- (A) ✅ **전역 + 프로젝트 동시.**
- (B) ✅ **자유텍스트 + 작성 가이드/템플릿 제공** — 작성은 사용자, 우리는 "쓰는 법"만 안내.
- (C) ✅ **리뷰 보정 포함** — 방향은 PR(의미 단락) 완성까지 적용되어야 하므로 리뷰에도 동일 적용.
- (D) ✅ **이슈별 전용 기능 없음** — 이슈 본문/검토 지시로 대체(이미 주입됨). 프롬프트에서
  이슈-레벨 오버라이드로 취급하도록 명시.
- (E) ✅ **명칭 = `Direction / 방향성`** (i18n 한/영) — 코드·문서·UI **전부 통일**.
  이전 코드네임 "Charter"는 폐기(내부/외부 이중 명칭 없음). 코드 식별자는 `direction`.
  대비 축: **Rules(규칙, skills) ↔ Direction(방향).**
- (F) ✅ **전역 저장 = userData 파일** `direction.md`(머신 단위) — config 필드 아님.
  코어는 경로 주입식으로 읽어 프로젝트 `.corral/DIRECTION.md`와 병합. 여러 프로젝트/config
  관통 전역이 목적.

## 14. UI 설명 문구 (요구사항 — 옵션에 반드시 노출)
skills만 쓰던 사용자는 "이미 skills가 있는데 왜 또?"라는 의문을 가장 먼저 가진다. 이 의문을
**옵션 안에서** 풀어주지 않으면 채택·활용이 안 된다. 아래를 옵션 화면에 넣는다(문구는 i18n).

- **라벨**: `Direction` / `방향성` (선택)
- **부제**: AI가 회사가 원하는 방향으로 판단·계획·리뷰하도록 하는 지침
- **짧은 설명(상시 노출)**: skills가 "어떻게 만들지(규칙)"라면, Direction은 "왜·어느
  방향으로(판단 기준)"입니다. 같은 문제라도 회사 우선순위와 트레이드오프 기준에 맞춰 풀도록
  계획과 리뷰에 반영됩니다.
- **펼침 "왜 필요한가요?"**: skills·profile은 규칙과 스택은 알려주지만, "안정 우선인지 속도
  우선인지", "새 의존성을 꺼리는지", "지금 이 제품이 뭘 가장 중요하게 여기는지" 같은 방향은
  담기지 않습니다. Direction을 한 번 적어두면 매 이슈마다 다시 말하지 않아도 일관되게 반영됩니다.
- **펼침 "그냥 skills에 적으면 안 되나요?"** ⭐(핵심): skills는 반드시 지켜야 하는 규칙(binding)
  으로 주입되고 리뷰가 위반을 지적합니다(WORKFLOW.md `Skills/conventions (REQUIRED)`,
  review 프롬프트 `Flag any violation`). 방향을 여기 넣으면 AI가 규칙처럼 강제해 더 나은 해법도
  비틀고, 리뷰가 방향을 "위반"으로 잡습니다. Direction은 규칙이 아니라 동률일 때의 기본 방향
  이며, 이슈의 정확성·요구사항이 항상 우선합니다 — 그래서 별도 층이 필요합니다.
- **대비 요약(한 줄)**: Rules 규칙(지켜야 함) ↔ Direction 방향성(판단 기준).
- **빈 화면 방지 스타터**(결정 B): `## 목적 / ## 우선순위 / ## 의사결정 원칙 / ## 하지 말 것`
  섹션 + 각 1~2줄 예시(예: "우선순위: 안정 > 기능 속도", "원칙: 새 의존성은 정당화 필요").
