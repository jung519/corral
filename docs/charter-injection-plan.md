# 작업계획서 — 조직 의도/방향 주입 (Charter Injection)

> 이 문서는 **계획서**입니다. 코드 작업은 포함하지 않습니다.

## 1. 목적
Corral이 "외부 도구를 잇는 오케스트레이터"를 넘어서, **AI가 회사의 목적·방향을 아는
직원처럼 이슈를 판단·계획·리뷰**하게 만든다. 같은 문제라도 "회사가 원하는 방향"으로
풀어내는 것 — 이것이 단순 연결형 도구와의 **차별점(해자)**.

## 2. 배경 / 현재 구조
현재 AI 추론에 들어가는 작업자 측 입력:
- `profile { language, stack }` — 얕은 맥락.
- **skills(reference repo)** — 컨벤션·디자인시스템. **이미 계획·구현·리뷰에 주입**됨
  (WORKFLOW "Skills/conventions REQUIRED", plan critique·review 프롬프트에서 compliance 확인).

빠진 것: **"의도/방향"** — 회사 목적, 제품 우선순위, 트레이드오프 판단 기준, 작업자 성향.
→ 계획의 방향·옵션, 리뷰의 보정에 **작업자 의도가 실리지 않음.**

## 3. 핵심 개념 — 3개 층 분리
| 층 | 답하는 질문 | 성격 | 상태 |
|---|---|---|---|
| skills | 어떻게 생겨야 하나 | 규칙(binding) | 있음 |
| profile | 어떤 스택/언어 | 맥락 | 있음(얕음) |
| **Charter(의도)** | **왜 / 어느 방향으로** | **판단 기준(guiding)** | **신설** |

Charter에 담는 것: ① 회사/제품 목적, ② 방향·우선순위(예: 안정 우선 vs 속도 우선),
③ 의사결정 원칙(트레이드오프 기준), ④ 작업자 성향(최소 변경, 의존성 회피 등).

## 4. 접근 확정: RAG ❌ / 주입 ✅
- 의도는 **작고 항상 관련** → 검색 불필요, **항상 주입**.
- skills와 동일한 주입 메커니즘, 단 프레이밍이 다름: **"comply(규칙 준수)" → "judge in line with(이 방향대로 판단)"**.
- RAG는 별개 층(큰 지식 코퍼스의 선택적 회수)으로, 이 기능 이후 필요 시.

## 5. 범위 — 저장 위치 (확정: 전역 + 프로젝트)
| 구분 | 위치 | 내용 |
|---|---|---|
| **전역(조직/작업자)** | 유저 설정 | 모든 프로젝트 공통 의도·성향 |
| **프로젝트** | `.corral/CHARTER.md` (또는 reference repo) | 이 레포의 목적·방향 (버전관리) |

**이슈별 방향은 전용 기능을 만들지 않는다(결정 D).** 특정 이슈만 다른 방향이 필요하면
사용자가 **이슈 본문**(최초 작성 시) 또는 **검토 단계 지시**(질문·논의 팝업의 수정요청/
지시승인)에 적는다 — 두 채널 모두 이미 추론에 주입된다. 트래커 전용 필드/라벨은
스키마 종속·중복이라 배제.

판단 우선순위(높→낮): **이슈 본문/검토 지시 > 프로젝트 Charter > 전역 Charter.**
단 **이슈의 정확성·요구사항이 항상 최우선**(Charter는 동률일 때의 기본 방향).

## 6. 주입 지점 (확정: 계획 + 리뷰)
- **계획**: kickoff · plan critique · consolidate 프롬프트에 Charter 블록 주입 →
  **방향과 생성 옵션이 의도를 반영**(옵션 축을 `보수/균형/적극`처럼 성향 축으로 제시).
  **+ 이슈 본문/검토 지시의 방향 힌트를 "그 이슈에 한해 Charter보다 우선 적용"하라고 명시**
  → 별도 UI 없이 이슈-레벨 오버라이드가 동작(결정 D 보강).
- **리뷰 (결정 C)**: 프로젝트 우선순위·방향에 맞춰 심각도·판단 보정(MVP면 NIT 완화,
  성숙 프로젝트면 엄격). **개발 방향은 리뷰까지 끝난 PR(의미 단락)에서 완성되므로 리뷰에도
  동일 Charter를 적용**한다.
- **구현(선택)**: 접근/스타일.

프레이밍 가드(프롬프트에 명시):
> "Charter는 **동률일 때의 기본 방향**이다. 이슈의 정확성·요구사항이 우선한다.
> skills(규칙)는 여전히 binding이다. Charter는 방향 판단의 기준이다."

## 7. 설계 개요 (구현 시)
- 저장: 전역 = 자유텍스트 설정값, 프로젝트 = `.corral/CHARTER.md`.
- UI: 셋업/환경설정에 Charter 입력란(자유텍스트, 결정 B).
- 주입: 프롬프트 빌더/워크플로 가이드에 Charter 블록 추가 — **기존 profile 주입과 동일 패턴**.
- 형식: **자유텍스트**(결정 B). 구조화 강제는 안 한다.

### 7.1 작성 가이드 (결정 B의 핵심)
내용은 결국 **사용자가 자유텍스트로 작성**한다. 우리는 "어떻게 쓰면 좋은지"만 안내한다 →
빈 화면을 주지 않도록 **도움말 + 플레이스홀더 + 스타터 템플릿**을 제공:
- 권장 섹션: **목적 / 우선순위 / 의사결정 원칙 / 하지 말 것**.
- 각 섹션 1~2줄 예시(예: "우선순위: 안정 > 기능 속도", "원칙: 새 의존성은 정당화 필요").
- 자유도는 유지하되, 좋은 Charter의 형태를 예시로 보여 품질을 끌어올린다.
- (Charter 품질이 결과 품질을 좌우하므로, 이 가이드 자체가 중요한 산출물.)

## 8. 단계
1. **전역 Charter(자유텍스트 + 작성 가이드/템플릿) → 계획 프롬프트 주입.**
2. **프로젝트 Charter(`CHARTER.md`) → 레포별 주입** + reference repo 연동.
3. **이슈 본문/검토 지시를 이슈-레벨 오버라이드로 인식**(프롬프트 한 줄 — 결정 D).
4. **옵션 생성에 방향 축** 반영(보수/균형/적극).
5. **리뷰 보정 적용**(결정 C) — 계획과 동일 Charter로 심각도·판단 보정.
6. **검증(eval)**: Charter on/off로 같은 이슈의 계획·옵션·리뷰 비교.

## 9. 1인 개발 적합성
- 거의 전부 **프롬프트·설정·UI** 작업 → **솔로 가능, 며칠 규모**. RAG 같은 인프라 없음.
- 가장 어려운 부분: **효과 입증(eval)** 과 **Charter 문구 튜닝**(과하지도 모자라지도 않게).

## 10. 리스크
- **과적재/모순**: 의도가 너무 많거나 충돌하면 편향·프롬프트 비대 → 간결 유지.
- **하드룰 오해**: 에이전트가 Charter를 규칙처럼 강제 적용 → "기본 방향" 명시로 완화.
- **정확성 침해**: 방향 때문에 옳은 해법을 비틀 위험 → "정확성·요구 우선" 가드.

## 11. 차별점 (왜 중요)
연결형 도구는 *"누가 일하나"* 만 정한다. Corral은 **"회사 의도대로 판단"** 을 인코딩 →
직원처럼 일관된 방향으로 계획·리뷰. 도구 교체로는 복제 안 되는 **제품 해자**.

## 12. 비범위 (이번 단계 밖)
- RAG / 임베딩 / 대형 지식 인덱싱 (별개 층, 추후).
- 실제 코드 작업 (이번 세션은 계획만).

## 13. 결정 (확정)
- (A) ✅ **전역 + 프로젝트 동시.**
- (B) ✅ **자유텍스트 + 작성 가이드/템플릿 제공** — 작성은 사용자, 우리는 "쓰는 법"만 안내.
- (C) ✅ **리뷰 보정 포함** — 방향은 PR(의미 단락) 완성까지 적용되어야 하므로 리뷰에도 동일 적용.
- (D) ✅ **이슈별 전용 기능 없음** — 이슈 본문/검토 지시로 대체(이미 주입됨). 프롬프트에서
  이슈-레벨 오버라이드로 취급하도록 명시.
