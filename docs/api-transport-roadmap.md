# API 트랜스포트 고도화 로드맵

`*:api` 트랜스포트(claude·gemini·gpt)는 공용 agentic 루프([`src/agent/api-loop.ts`](../src/agent/api-loop.ts))
위에서 동작합니다. 현재는 `bash` 도구 하나만으로 코딩을 수행해, 성숙한 CLI(claude-code·
gemini-cli·codex)에 비해 정밀도·안전성·효율이 부족합니다. 이 문서는 그 격차를 Phase로
나눠 메우는 계획입니다.

핵심 원칙: **개선의 대부분은 공용 엔진에서 처리 → 3개 provider가 한 번에 좋아짐**
(캐싱·가격표만 provider별). 검증은 전 Phase 공통으로 가짜 fetch/클라이언트 단위테스트로
하므로 실제 API 키가 필요 없습니다.

## 한눈에 보기

| Phase | 목표 | 규모 | 리스크 | 의존성 |
|---|---|---|---|---|
| 1 | 정밀 도구 세트(read/write/edit/ls/grep) + 페이지네이션 | L | 중 | — |
| 2 | 안전성: 샌드박스·명령 정책·allowedTools | M | 중 | 1 |
| 3 | 비용 집계 + 예산 상한 | M | 하 | — |
| 4 | 회복력: 재시도·백오프·레이트리밋 | S~M | 하 | — |
| 5 | 효율: 프롬프트 캐싱 + 컨텍스트 자동 압축 | L | 중상 | 1 |
| 6(선택) | 스트리밍 UX | L | 중 | — |

진행 순서 원칙: **체감 품질 → 안전성 → 정확성 → 신뢰성 → 효율**.

> 진행 상태: Phase 1~6 전부 완료 ✅

---

## Phase 1 — 정밀 도구 세트 + 읽기 페이지네이션 ⭐
`bash` 단일 도구를 CLI급 전용 도구로 확장. 체감 품질이 가장 큰 단계.

- 도구 추가: `read`(offset/limit·줄번호), `write`, `edit`(정확 문자열 치환), `ls`, `grep`.
  `bash`는 빌드/테스트/git용으로 유지.
- edit 안전 규칙: 대상 0개→"없음" 에러, 2개↑→"모호함" 에러, 1개만 치환. read 안 한
  파일은 edit 거부(read-before-edit) — 엔진에 파일 읽음-상태 추적.
- 출력 위생: `read`는 페이지네이션(기본 2000줄 + 잘림·총줄수 표시), `bash` 출력은 뭉텅
  컷 대신 head+tail+생략표시.
- 파일: `api-loop.ts`(도구 정의·디스패처·파일상태). 클라이언트 3개는 `ToolDef[]`를 이미
  매핑하므로 거의 무수정.
- 검증: 도구 디스패치, edit 4케이스(성공/없음/모호/미read), read 페이지네이션, 회귀.

## Phase 2 — 안전성 (샌드박스·명령 정책·allowedTools)
현재 bash가 무샌드박스·무승인 즉시 실행되고 `allowedTools`도 무시됨. 로컬 백엔드 위험.

- `spec.allowedTools` 실제 적용(도구 목록 필터).
- 명령 정책: 위험 명령 기본 차단 또는 allowlist, 워크스페이스 경로 이탈 방지.
- (선택) 위험 명령 시 채널로 사람 승인 요청.
- UI: API 모드는 도커 백엔드 권장/기본, 로컬+API엔 경고.
- 파일: `api-loop.ts`, `Wizard.svelte`.
- 검증: 정책 차단/허용, allowedTools 필터, 경로 이탈 차단.

## Phase 3 — 비용 집계 + 예산 상한
현재 `costUsd=0`이라 `maxBudgetUsd`가 무력. 무한방지는 `maxTurns`뿐.

- provider×model 가격표(입력/출력 1M토큰당), config로 덮어쓰기 가능.
- 각 클라이언트가 usage→costUsd 계산.
- 루프가 누적 비용 추적 → `maxBudgetUsd` 초과 시 `error:'budget'`로 중단(폴백 대상).
- 파일: `api-loop.ts`, 3개 클라이언트, 가격표 모듈/스키마.
- 검증: 비용 계산 정확도, 예산 초과 중단.

## Phase 4 — 회복력 (재시도·백오프·레이트리밋)
일시적 5xx/429에 그 턴이 그냥 죽음.

- 재시도 가능 에러(429·5xx·네트워크) 지수 백오프 + `Retry-After` 존중, 최대 횟수 제한.
- 파일: `api-loop.ts`의 `send` 호출 래퍼(공통).
- 검증: 1회 실패 후 성공, 한도 초과 시 포기.

## Phase 5 — 효율 (프롬프트 캐싱 + 컨텍스트 자동 압축)
매 턴 전체 기록 재전송 → 토큰·비용 과다, 긴 작업은 컨텍스트 초과로 `crashed`.

- 프롬프트 캐싱: Anthropic `cache_control`, OpenAI 자동, Gemini context cache(여력 시).
- 자동 압축: 컨텍스트 한도 근접 시 오래된 턴을 요약 다이제스트로 치환(최근 K턴·시스템·
  과제는 보존).
- 파일: `api-loop.ts`(압축), 클라이언트(캐싱 헤더).
- 검증: 캐시 필드 존재, 압축 트리거 + 최근 턴 보존.

## Phase 6 (선택) — 스트리밍 UX
턴 단위 비스트리밍 → SSE로 토큰 점진 출력 + 도구 diff 미리보기.

- 각 클라이언트 `send`를 스트리밍 엔드포인트로, `text` 델타·풍부한 `tool_use` 이벤트.
- 규모 큼(send 재작성), 우선순위 낮음.

---

## 추천 묶음
**Phase 1 → 2 → 3**이 "실용 → 안전 → 통제"의 최소 묶음. 4·5는 운영하며 필요할 때,
6은 여유 시.
